## Abstract

This DEP describes the Decent Enhancement Proposal (DEP) process for evolving the DecentChat protocol. DEPs are design documents providing information to the DecentChat community, describing new features, protocol changes, or process improvements.

## Motivation

As DecentChat grows, we need a formal mechanism for:

- **Coordinating protocol changes** across multiple implementations
- **Documenting design decisions** and technical rationale
- **Community participation** in protocol evolution
- **Backward compatibility** tracking and migration planning
- **Clear governance** while maintaining decentralization

## DEP Types

### 1. Core Protocol
Changes to the fundamental protocol (encryption, message format, sync, identity).

**Examples:**
- New cryptographic primitives
- Message format versioning
- Sync protocol changes

### 2. Transport
Changes to peer discovery, signaling, or network transport.

**Examples:**
- Signaling server discovery (PEX)
- New transport mechanisms (WebTransport, libp2p)
- NAT traversal improvements

### 3. Application
Client features that don't require protocol changes but benefit from standardization.

**Examples:**
- Slash command standards
- Emoji picker behavior
- Notification policies

### 4. Process
Governance, development process, or meta-protocol changes.

**Examples:**
- This document (DEP-000)
- Security disclosure process
- Release versioning scheme

## DEP Workflow

### Statuses

1. **Draft** — Work in progress, under discussion
2. **Proposed** — Ready for review, seeking feedback
3. **Active** — Accepted, implementation in progress
4. **Final** — Shipped and stable
5. **Deprecated** — Superseded by newer DEP
6. **Rejected** — Considered but not accepted

### Process

1. **Author drafts DEP** using the template (see below)
2. **Submit PR** to `decent-chat/specs/deps/`
3. **Community discussion** via GitHub issues/PR comments
4. **Revision** based on feedback
5. **Alex approves** → Status moves to **Proposed**
6. **Implementation PR** links to DEP
7. **After merge** → Status moves to **Active**
8. **After release + stability** → Status moves to **Final**

### Numbering

- DEPs are numbered sequentially: `DEP-001`, `DEP-002`, etc.
- Numbers are assigned when the PR is opened (first-come basis)
- Use `DEP-XXX` as placeholder in draft before PR

## Governance

- **Alex** is the benevolent dictator — final say on **Proposed → Active** transitions
- **Anyone** can submit a DEP (Draft → Proposed requires Alex approval)
- **Rough consensus** is preferred; Alex decides if consensus isn't reached
- **Transparency** — all discussion happens in public (GitHub PRs/issues)

## Mandatory vs Optional DEPs

- **Core Protocol** DEPs are **mandatory** for compatibility
- **Transport** DEPs may be optional (fallback to base protocol)
- **Application** DEPs are **recommendations** (clients can differ)
- **Process** DEPs are procedural (not code)

## Updating DEPs

- **Draft/Proposed** DEPs can be freely edited
- **Active** DEPs should only receive clarifications (not breaking changes)
- **Final** DEPs are immutable (use new DEP with `Replaces:` field to supersede)
