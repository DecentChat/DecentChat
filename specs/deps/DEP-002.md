# DEP-002: Peer Exchange (PEX) for Signaling Server Discovery


## Abstract

Enable peers to share known signaling servers with each other, creating a self-healing network that doesn't depend on a single signaling server. Inspired by BitTorrent's Peer Exchange (PEX).

## Motivation

**Current problem:**
- Invite links hardcode a single signaling server
- If that server goes down, the workspace becomes unreachable
- No mechanism for discovering alternative servers

**With PEX:**
- Peers learn backup servers from each other
- Network self-heals if primary server fails
- Organic decentralization without central coordination

**Example:**
1. You join via `wss://signal1.com`
2. I share `wss://signal2.com` in our handshake
3. Your client adds it to known servers
4. If `signal1` goes down, you auto-reconnect via `signal2`
5. Network stays connected

## Specification

### Handshake Extension

Add `signalingServers` field to peer handshake:

```typescript
interface HandshakeMessage {
  // ... existing fields ...
  signalingServers?: Array<{
    url: string
    lastSeen: number      // timestamp
    successRate: number   // 0.0 - 1.0
    latency?: number      // ms (optional)
  }>
}
```

### Periodic Exchange

Peers broadcast server list via heartbeat:

```typescript
interface PeerExchangeMessage {
  type: 'peer-exchange'
  signalingServers: Array<{
    url: string
    lastSeen: number
    successRate: number
  }>
}
```

**Frequency:** Every 5 minutes (piggybacked on existing heartbeats)

### Client Behavior

1. **On connect:** Send top 3-5 servers in handshake
2. **On heartbeat:** Broadcast updated server list (if changed)
3. **On receive:** Merge into local known-servers list
4. **On reconnect:** Try servers in ranked order:
   - Primary (from invite) first
   - Then by `successRate × (1 / latency)`
   - Exponential backoff on failures

### Server Ranking

```typescript
function rankServer(server: SignalingServer): number {
  const age = Date.now() - server.lastSeen
  const recency = Math.exp(-age / (7 * 24 * 3600 * 1000)) // 7-day decay
  const reliability = server.successRate
  const speed = 1 / (server.latency || 1000)
  
  return recency * 0.3 + reliability * 0.5 + speed * 0.2
}
```

### Persistence

Store known servers in IndexedDB:

```typescript
interface KnownServersDB {
  workspaceId: string
  servers: Array<{
    url: string
    addedAt: number
    lastSeen: number
    successRate: number
    totalAttempts: number
    successfulAttempts: number
    avgLatency: number
  }>
}
```

## Rationale

**Why not just use DNS for server discovery?**
- Centralized (single point of failure)
- Requires infrastructure
- PEX is purely peer-driven

**Why broadcast via heartbeat instead of on-demand?**
- Lower latency (don't wait for failure to learn alternatives)
- Spreads load (gradual diffusion instead of thundering herd)

**Why limit to top 3-5 servers?**
- Prevent bandwidth waste from huge server lists
- Focus on high-quality servers
- 5 is enough redundancy for resilience

**Alternatives considered:**
- **DHT-based discovery** — too complex, high overhead
- **Gossip protocol** — similar to PEX, but more rounds
- **Central directory** — defeats purpose of decentralization

## Backward Compatibility

**Migration:**
- PEX is **optional** — clients without it still work
- Old clients ignore `signalingServers` field (unknown fields are dropped)
- New clients detect PEX support via protocol version handshake

**Protocol version:**
- Advertise `pex: true` in capabilities
- If peer lacks PEX, don't send exchange messages

**Pseudocode example:**

```typescript
class PeerExchange {
  private knownServers: Map<string, ServerStats> = new Map()
  
  // On handshake, share top servers
  getHandshakeServers(): SignalingServer[] {
    return Array.from(this.knownServers.values())
      .sort((a, b) => rankServer(b) - rankServer(a))
      .slice(0, 5)
      .map(s => ({ url: s.url, lastSeen: s.lastSeen, successRate: s.successRate }))
  }
  
  // On receive, merge into local knowledge
  mergeReceivedServers(servers: SignalingServer[]) {
    for (const s of servers) {
      const existing = this.knownServers.get(s.url)
      if (!existing || s.lastSeen > existing.lastSeen) {
        this.knownServers.set(s.url, { ...s, addedAt: Date.now() })
      }
    }
    this.pruneOldServers() // Remove servers not seen in 30 days
  }
  
  // On reconnect, try servers by rank
  async connect(workspaceId: string): Promise<Peer> {
    const ranked = this.getRankedServers()
    for (const server of ranked) {
      try {
        const peer = await connectViaServer(server.url, workspaceId)
        this.recordSuccess(server.url)
        return peer
      } catch (err) {
        this.recordFailure(server.url)
      }
    }
    throw new Error('All signaling servers failed')
  }
}
```

## Security Considerations

**Malicious server injection:**
- Attacker shares `evil.com` as signaling server
- Your client tries it, attacker MitM's connections

**Mitigation:**
- Only accept servers from **already-trusted peers** (post-handshake)
- TLS required for all signaling servers (reject `ws://`, only `wss://`)
- Peer public key verification prevents MitM after initial connect

**DoS via server spam:**
- Attacker floods you with 10,000 fake servers

**Mitigation:**
- Cap known servers at 50 per workspace
- Rate-limit PEX messages (max 1 per 5 minutes per peer)
- Prune servers not seen in 30 days

**Privacy leak:**
- Sharing server list reveals your connection history

**Acceptable:** Server URLs are already semi-public (in invite links). Not a significant leak.

## Test Vectors

### Scenario 1: Primary server down, fallback succeeds
```
Known servers: [signal1.com (down), signal2.com (up), signal3.com (up)]
Connect attempt: signal1 → fail, signal2 → success
Result: Connected via signal2
```

### Scenario 2: Learn new server from peer
```
Your servers: [signal1.com]
Peer shares: [signal1.com, signal2.com]
Merge: Now you know [signal1.com, signal2.com]
Next reconnect: Try both
```

### Scenario 3: Server ranking
```
Servers:
- signal1: lastSeen=now, success=95%, latency=50ms
- signal2: lastSeen=1d ago, success=99%, latency=200ms
- signal3: lastSeen=now, success=80%, latency=30ms

Ranking: signal1 > signal3 > signal2
(recency + reliability + speed)
```

## Performance Impact

**Bandwidth:**
- Handshake: +200 bytes (5 servers × 40 bytes)
- Heartbeat: +200 bytes every 5 minutes
- **Total:** ~0.7 KB/hour per peer (negligible)

**Storage:**
- IndexedDB: ~2 KB per workspace (50 servers × 40 bytes)

**Latency:**
- Reconnect: +50ms per failed server attempt
- With 3 servers ranked, max +150ms to find working server

**Benefit:** Network stays connected even if servers fail → **infinite uptime improvement**
