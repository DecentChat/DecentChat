## Abstract

Replace the current Merkle tree-based sync protocol with Negentropy set reconciliation for efficient message synchronization between peers. Negentropy achieves O(differences) complexity instead of O(log n), dramatically reducing sync overhead when message sets are mostly identical.

## Motivation

**Current problem:**
- Merkle tree sync: O(log n) round trips to find differences
- Bandwidth waste when syncing 10,000 messages but only 5 are new
- Long reconnection times after extended offline periods

**With Negentropy:**
- O(differences) — only transmit data proportional to actual changes
- Sub-second sync for weeks of offline messages
- Proven in Nostr ecosystem (Holepunch implementation)

## Specification

### Protocol Flow

1. **Initiator sends Negentropy query:**
   ```typescript
   interface NegentropyQuery {
     type: 'negentropy-query'
     channelId: string
     ranges: Array<{
       start: string  // timestamp or ID
       end: string
       fingerprint: string // set hash
     }>
   }
   ```

2. **Responder compares fingerprints, returns differences:**
   ```typescript
   interface NegentropyResponse {
     type: 'negentropy-response'
     channelId: string
     have: string[]      // message IDs responder has, initiator missing
     need: string[]      // message IDs initiator has, responder missing
     nextQuery?: NegentropyQuery  // if more rounds needed
   }
   ```

3. **Peers exchange missing messages**

4. **Repeat until fingerprints match**

### Integration Points

- **Per-channel sync** — each channel maintains its own Negentropy state
- **Incremental updates** — new messages update local fingerprint
- **Backward compatible** — fallback to Merkle tree if peer doesn't support Negentropy

### Data Structures

```typescript
interface NegentropyState {
  channelId: string
  items: Array<{
    id: string
    timestamp: number
  }>
  fingerprint: string
}
```

## Rationale

**Why not stick with Merkle trees?**
- Merkle: O(log n) rounds × O(k) messages per round = slow for large sets
- Negentropy: Single round for most syncs

**Why not just send everything?**
- Bandwidth explosion (10K messages × avg 2KB = 20MB)
- Negentropy: ~500 bytes for fingerprints + only missing messages

**Alternatives considered:**
- CRDT sync (too complex, requires causal ordering)
- Bloom filters (false positives, multiple rounds)
- Range-based sync (requires sorted IDs, fragile)

## Backward Compatibility

**Migration path:**
1. Add Negentropy support alongside Merkle tree
2. Peers advertise capabilities in handshake
3. If both support Negentropy → use it
4. If one lacks Negentropy → fallback to Merkle tree
5. Eventually deprecate Merkle tree in v2.0

**Protocol version:**
- Bump to `v1.1` with `negentropy` capability flag
- No breaking changes for v1.0 clients

**Known limitations** (2 edge cases, not production-relevant):
  1. **Sparse diffs** (every Nth item missing) - Requires deep subdivision for perfectly uniform distribution. Real message sync has temporal clustering, not uniform distribution.
  2. **Same-timestamp ID conflicts** - When two items share the exact same timestamp but have different IDs, fingerprint-based sync cannot distinguish them. Solution: Use unique message IDs with sufficient timestamp resolution (milliseconds).

## Security Considerations

**Fingerprint collisions:**
- Use SHA-256 for set fingerprints (collision probability negligible)
- If collision occurs, fallback to full sync

**DoS via fake fingerprints:**
- Rate-limit sync requests per peer
- Disconnect peers sending invalid fingerprints repeatedly

**Privacy:**
- Fingerprints reveal message count and distribution
- Not a concern for group channels (already visible)
- For DMs, fingerprints leak message timing (acceptable tradeoff)
